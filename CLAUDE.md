## Overview

DHIS2 FlexiPortal is a monorepo containing a public-facing portal for unauthenticated access to DHIS2 data, plus a DHIS2 custom app (manager) for configuring portal content. The package manager is **pnpm** (not yarn ŌĆö the README is outdated). Build orchestration uses **Turborepo**.

## Monorepo Structure

- `apps/portal/` ŌĆö Next.js 16 public portal (SSR, app router, React 19)
- `apps/manager/` ŌĆö DHIS2 custom app built with `@dhis2/cli-app-scripts`, TanStack Router for routing
- `apps/docs/` ŌĆö Docusaurus documentation site
- `packages/shared/` ŌĆö Shared Zod schemas, constants, utility functions, and visualization components used by both apps
- `services/data-service/` ŌĆö Bun/Express API service that syncs data from DHIS2 to a local store via RabbitMQ; uses Prisma with PostgreSQL

## Commands

### Root-level (run from repo root)

```bash
pnpm install          # Install all dependencies
pnpm dev              # Run all apps concurrently
pnpm dev:portal       # Run only the portal (localhost:3000)
pnpm dev:manager      # Run only the manager (localhost:3001)
pnpm build            # Build all apps via Turborepo
pnpm lint             # Auto-fix lint issues across all packages
pnpm lint:check       # Check lint without fixing
pnpm format           # Format with Prettier
pnpm format:check     # Check formatting
pnpm check-types      # TypeScript type check across all packages
pnpm test             # Run vitest unit tests (apps/* and packages/*)
```

### Per-app (using pnpm filter)

```bash
pnpm --filter portal build
pnpm --filter manager build
pnpm --filter manager publish:app-hub   # Publish to DHIS2 App Hub
```

### Running a single test

```bash
pnpm --filter portal test -- --run src/path/to/file.test.ts
pnpm --filter shared test -- --run src/path/to/file.test.ts
```

### Portal E2E tests

```bash
pnpm portal test:e2e        # Cypress headless
pnpm portal test:e2e-open   # Cypress with UI
```

### Manager E2E tests

```bash
pnpm manager test:e2e
pnpm manager test:e2e-open
```

### Data service (run from `services/data-service/`)

```bash
pnpm dev              # Run both API and worker concurrently
pnpm dev:api          # Bun API server only
pnpm dev:worker       # RabbitMQ worker only
pnpm prisma:generate  # Regenerate Prisma client
pnpm prisma:migrate   # Run DB migrations
pnpm prisma:studio    # Open Prisma Studio
```

## Environment Variables

**Portal** (`apps/portal/.env`):

- `DHIS2_BASE_URL` ŌĆö URL of the DHIS2 instance
- `DHIS2_BASE_PAT_TOKEN` ŌĆö Personal Access Token for server-side API calls
- `NEXT_PUBLIC_CONTEXT_PATH` ŌĆö Optional subpath for non-root deployments

**Manager** (`apps/manager/.env`):

- `DHIS2_PROXY_URL` ŌĆö DHIS2 instance URL for dev proxy

**Data service** (`services/data-service/.env`):

- `DHIS2_BASE_URL`, `DHIS2_PAT` ŌĆö DHIS2 connection
- `DATA_SERVICE_PORT`
- `RABBITMQ_HOST`, `RABBITMQ_USER`, `RABBITMQ_PASS`, `RABBITMQ_URI` (optional), `RABBITMQ_PREFETCH_COUNT` (optional)

## Architecture

### Portal (Next.js App Router)

The portal is a **server-rendered Next.js app**. All DHIS2 data fetching happens server-side using `dhis2HttpClient` (a server-only HTTP client that injects the PAT token). The client never has access to the PAT token.

- **Route**: `src/app/modules/[...module]/page.tsx` ŌĆö central catch-all; last path segment is the `moduleId`. A `/details/{resourceId}` sub-path is also handled for resource detail views.
- **API proxy**: `src/app/api/[...path]/route.ts` ŌĆö whitelisted DHIS2 API proxy for client-side chart/map rendering (only allows `analytics`, `legendSets`, `organisationUnits`, `geoFeatures`, `tokens/google`). Additional internal routes: `/api/documents`, `/api/icons`, `/api/info`.
- **Config storage**: Portal configuration is read from DHIS2 Datastore at runtime; `reactStrictMode` is disabled in `next.config.ts`; `NEXT_PUBLIC_CONTEXT_PATH` affects both runtime (`basePath`) and the Turborepo build cache.

### Manager (DHIS2 Custom App)

Built with `@dhis2/cli-app-scripts` (Vite-based). Uses **TanStack Router** (file-based routing in `src/modules/`). Uses **hash-based routing** (`createHashHistory()`). Configuration is saved to DHIS2 Datastore. The manager writes module configs that the portal then reads.

- Routes are auto-generated into `src/routeTree.gen.ts` ŌĆö do not edit this file manually
- File-name conventions: `__root.tsx` = root, `_provider.tsx` = layout wrapper (creates `<Outlet>`), `index.lazy.tsx` = lazy-loaded leaf, `$paramName` = dynamic segment
- Datastore namespaces: `hisptz-public-portal` (keys: `appearance`, `menu`, `metadata`); `hisptz-public-portal-modules` (per-module configs keyed by module ID); `hisptz-public-data-service-config`

### Shared Package (`@packages/shared`)

Imported via workspace alias. Exports:

- `@packages/shared/schemas` ŌĆö Zod schemas for all data types (modules, appearance, visualizations, etc.)
- `@packages/shared/constants` ŌĆö Datastore namespace constants, colors, screen sizes
- `@packages/shared/components` ŌĆö Shared React visualization components (charts, maps, tables)
- `@packages/shared/visualizations` ŌĆö Visualization-specific subpath export
- `@packages/shared/utils` ŌĆö Utility functions
- `@packages/shared/hooks` ŌĆö Shared React hooks

The package is listed under `transpilePackages` in portal's `next.config.ts` because it uses the workspace alias.

The `ModuleType` enum and all module config schemas live here and are the single source of truth for both apps.

### Data Service

Optional background service for scenarios where the portal needs to serve data from a disconnected or replicated source. Uses:

- **Express 5** as the HTTP API
- **RabbitMQ** for async job queuing (data download, metadata processing, upload)
- **Prisma + PostgreSQL** for tracking run status
- **Bun** as the runtime (not Node.js)

## Key Conventions

- All schemas are defined with **Zod** in `packages/shared/src/schemas/` and inferred as TypeScript types
- Form handling uses **react-hook-form** with `@hookform/resolvers` for Zod integration
- Data fetching in both apps uses **TanStack Query** (`@tanstack/react-query`)
- Portal UI: **Mantine** component library + **Tailwind CSS**
- Manager UI: **DHIS2 UI** components (`@dhis2/ui`) + **Tailwind CSS**
- Commits follow **Conventional Commits** (enforced via `lint-staged` + husky); releases are automated with `semantic-release`
- Unit tests use **Vitest** with files named `*.test.ts(x)`
